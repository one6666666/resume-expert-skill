# AI 简历专家 | 简历生成与优化

## 概述

本 Skill 提供从零生成简历、JD 驱动优化、ATS 兼容检测到面试准备的全流程能力。一次输入，直接输出可投递的专业简历（HTML + DOCX）。

核心原则：
- **拒绝千篇一律**：按岗位类型动态决定技能维度和表达重点
- **基于真实经历**：不编造数据，信息不足时主动追问
- **ATS 友好**：关键词自然嵌入，结构清晰可解析
- **行动导向**：不只给建议，直接生成可投递的成品


## 使用场景与决策树

根据用户输入，判断进入哪种模式：

```
用户输入
  ├─ 有简历 + 有 JD → 模式 A: JD 驱动优化
  ├─ 有简历 + 无 JD → 模式 B: 简历审计与提升
  ├─ 无简历 + 有 JD → 模式 C: 从 JD 反向生成
  ├─ 无简历 + 无 JD → 模式 D: 信息收集 → 从零生成
  ├─ 只要面试建议   → 模式 E: 面试准备
  └─ 只要格式转换   → 模式 F: 格式输出
```


## 模式 A — JD 驱动优化（最常用）

### A.1 JD 解析

拿到 JD 后，先提取以下结构化信息：

1. **岗位核心职责**（3-5 条）：公司希望你入职后做什么
2. **必须技能**（Must-have）：JD 中明确要求的硬技能
3. **加分技能**（Nice-to-have）：优先项或偏好
4. **高频关键词**（5-10 个）：JD 中反复出现的词，必须在简历中出现
5. **软技能隐含要求**：从 JD 措辞推断（如"跨部门协作"→沟通能力）

输出格式：
```
## JD 解析结果

### 目标岗位：XXX
### 核心职责
1. ...
2. ...

### 必须技能关键词
Java, Spring Boot, MySQL, Redis, Kafka

### 加分技能关键词
Docker, K8s, 微服务

### 高频关键词（必须覆盖）
高并发, 分布式, 性能优化, 系统设计, 可用性
```

### A.2 简历诊断

对照 JD 解析结果，诊断现有简历：

1. **关键词覆盖率**：列出覆盖的和缺失的关键词
2. **经历匹配度**：每条经历与 JD 的相关性评估
3. **量化缺失**：哪些描述缺少数据支撑
4. **表达问题**：职责描述 vs 成就描述
5. **结构问题**：模块顺序是否需要调整

输出格式：
```
## 简历诊断报告

### 📊 整体评分
- JD 关键词覆盖率：60%（15/25）
- 经历相关性：⭐⭐⭐
- 量化表达：⭐⭐
- 结构清晰度：⭐⭐⭐⭐

### ❌ 缺失关键词
Docker, K8s, 微服务架构, 容灾设计

### ⚠️ 需要重写的条目
1. "负责系统开发" → 缺少技术栈和成果
2. "参与数据库设计" → 缺少规模和优化结果

### ✅ 优势条目（保持）
1. "QPS 从 500 提升至 5000" → 量化好，保留
```

### A.3 执行优化

根据诊断结果，逐条重写。对每条需要修改的经历：

1. 保持原始事实不变
2. 加入缺失关键词（自然嵌入，不堆砌）
3. 补充量化数据（如无数据，标记 ⚠️ 让用户补充）
4. 将"做了什么"改成"做成了什么"
5. 使用格式：**加粗概括 + 冒号 + 详细描述**

示例：
```
改写前：负责公司后端服务开发与维护
改写后：负责核心业务后端服务的架构设计与迭代开发，日均处理请求 50万+，系统可用性 99.9%
```


## 模式 B — 简历审计与提升

无需 JD，对简历做通用质量提升。

### B.1 30秒结论

用一句话总结简历最大问题。

### B.2 分维度评分（每项 ⭐1-5）

| 维度 | 评分 | 说明 |
|------|------|------|
| 内容完整度 | ⭐⭐⭐ | ... |
| 表达专业度 | ⭐⭐⭐ | ... |
| 量化程度 | ⭐⭐ | ... |
| 结构清晰度 | ⭐⭐⭐⭐ | ... |
| ATS 兼容性 | ⭐⭐⭐ | ... |

### B.3 逐项修改建议

对每条经历给出改写版本。

### B.4 补充清单

列出需要用户补充的信息。


## 模式 C — 从 JD 反向生成

用户只有 JD，没有简历。先询问关键信息，再生成。

### C.1 信息收集（必问）

1. 姓名、联系方式、所在地
2. 教育背景（学校、专业、学历、时间）
3. 工作经历（公司、职位、时间段、主要职责和成果）
4. 项目经验（项目名、你的角色、技术栈、成果）
5. 技能清单

### C.2 生成策略

根据 JD 反向定制：
- 将与 JD 最相关的经历放在最前面
- 技能模块优先展示 JD 所需技能
- 项目描述中嵌入 JD 高频关键词


## 模式 D — 从零生成

用户没有任何材料。用对话式收集信息：

```
请告诉我以下信息，我帮你生成简历：

1. 你应聘什么岗位？有没有目标 JD？
2. 你的教育背景？
3. 你的工作经历？（公司、职位、时间、主要做了什么、取得了什么成果）
4. 你做过哪些项目？（项目名称、你的角色、技术栈、成果）
5. 你掌握哪些技能？
```

信息不足时继续追问，直到信息充分后再生成。


## 模式 E — 面试准备

基于 JD 和优化后的简历，生成面试准备材料：

1. **高频考点预判**（3-5 个）：根据岗位类型预判必问问题
2. **亮点故事线**（3 个）：简历中最值得展开的亮点
3. **弱点预判**：简历中可能被追问的薄弱点及应对
4. **反问建议**（3 个）：面试结束时可以反问面试官的问题


## 模式 F — 格式输出

### 输出 HTML 简历

读取 `assets/resume_template.html` 模板，使用 Mustache 风格变量替换：

```html
变量列表：
{{name}}           — 姓名
{{target_position}} — 目标岗位
{{years}}          — 工作年限
{{email}}          — 邮箱
{{phone}}          — 电话
{{location}}       — 所在地
{{github}}         — GitHub（可选区块）
{{linkedin}}       — LinkedIn（可选区块）
{{summary}}        — 个人简介（可选区块）
{{skills}}         — 技能列表（数组）
{{experience}}     — 工作经历（对象数组，每项含 role/company/date/details）
{{projects}}       — 项目经验（对象数组，每项含 name/tech/description）
{{education}}      — 教育背景（对象，含 degree/school/date/edu_note）
```

生成步骤：
1. 读取模板 `assets/resume_template.html`
2. 将简历数据填入 HTML
3. 保存为 `.html` 文件
4. 告知用户可在浏览器打开并用 ⌘+P 打印为 PDF

### 输出 DOCX 简历

使用 `scripts/generate_docx.py` 脚本：

```
# 1. 将简历数据写入临时 JSON 文件
# 2. 安装依赖（如未安装）
pip install python-docx

# 3. 运行脚本
python scripts/generate_docx.py <output_path.docx> <json_data.json>
```

JSON 数据结构见脚本顶部注释。

### 输出 ATS 纯文本版

生成适合直接粘贴到招聘系统的纯文本版本：
- 使用 `=====` 分隔各模块
- 所有内容左对齐
- 技能用 `、` 分隔
- 不使用特殊符号

支持的岗位类型（详见 references/job_types.md）：
1. 后端工程师
2. 前端工程师
3. 算法/ML/AI 工程师
4. 产品经理
5. AI 产品经理
6. 数据分析师
7. 运营（用户/内容/增长）
8. 设计师（UI/UX/视觉）
9. 测试/QA 工程师
10. 市场/品牌
11. 财务/金融

**重要**：在生成简历前，参考 `references/job_types.md` 中对应岗位的技能维度，确保：
- 技能模块包含该岗位的核心关键词
- 项目描述中使用该岗位的专业术语
- 工作经历突出该岗位最看重的成果类型


## 简历内容质量标准

### 每一条工作经历必须满足
1. **具体动作** — 明确做了什么（设计、开发、优化、主导、推动）
2. **技术/方法** — 用了什么工具、技术、方法论
3. **量化结果** — 产生了什么可衡量的影响
4. **业务价值** — 对团队/公司/用户有什么价值

### 量化挖掘规则
优先提取以下类型数据：
- **规模类**：QPS、DAU、数据量、用户数、交易额
- **提升类**：性能提升 X%、效率提升 X 倍、成本降低 X%
- **时间类**：提前 X 周交付、响应时间从 A 优化到 B
- **质量类**：可用性 99.X%、Bug 率降低 X%

如果用户未提供数据，主动追问【请在生成简历前追问】。

### 禁止事项
- ❌ 编造不存在的数据
- ❌ 夸大技术能力
- ❌ 使用空洞形容词（"认真负责""学习能力强"）
- ❌ 套用与经历无关的模板话术
- ❌ 一份简历对应所有岗位


## ATS 兼容性规则

### 结构规范
- 使用标准模块标题：工作经历、教育背景、技能、项目经验、个人简介
- 工作经历按时间倒序排列
- 不使用表格（ATS 可能解析错误）
- 不使用图片、图表、图标
- 不使用页眉页脚存放关键信息

### 关键词策略
- JD 中的关键词必须自然出现在以下位置：
  - 技能模块（直接列出）
  - 工作经历描述（融入具体成果）
  - 项目经验（技术栈标注）
- 不要过度堆砌（每 100 字出现 3-5 个行业关键词即可）

### 格式要求
- 字体：中文使用微软雅黑/苹方，英文使用 Arial/Helvetica
- 字号：正文 10-11pt，标题 13-16pt
- 页边距：上下 1.5-2cm，左右 2-2.5cm
- 文件格式：优先 PDF，其次 DOCX
- 文件命名：`姓名_岗位_简历.pdf`

